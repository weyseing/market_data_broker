STACK
- Python 3.11+, asyncio
- websockets, pydantic, mcp (FastMCP), structlog, pyyaml
- Tests: pytest, pytest-asyncio
- Single Docker container, in-memory state only

TOPIC NAMING
  coinbase.{channel}.{product_id}    e.g. coinbase.ticker.BTC-USD
  Channels in scope: ticker, matches, level2_batch
  Venue prefix leaves room to add binance.* later without breaking changes.

BACKPRESSURE
  Bounded asyncio.Queue per consumer (default 1000).
  Overflow → drop-oldest + increment dropped_messages counter (visible in status).
  Sustained overflow → disconnect consumer with reason.

           Decisions worth flagging:

Step 3  — In-memory pub/sub bus                                             [DONE]
  bus.py: InMemoryBus.publish / subscribe / unsubscribe.
  Bounded per-consumer queues, drop-oldest, drop counters.
  Verify: multi-consumer fan-out + slow-consumer + dynamic sub/unsub tests pass.
  Outcome: 28 new tests green; full suite 83/83; ruff clean.
           Decisions worth flagging:
           - Subscription is the consumer handle (async-iterable, has its own
             queue, dropped_messages counter). Bus owns routing tables;
             Subscription delegates add_topic / remove_topic / close back to it.
           - publish() is async-shaped but never yields — fan-out is atomic
             w.r.t. other coroutines, so no locks. Keeps callers consistent
             (`await bus.publish(msg)`).
           - Drop-oldest implemented as get_nowait()+put_nowait() with the
             counter incremented per drop. Bus does NOT close unhealthy subs —
             that policy belongs in the registry (Step 4).
           - Subscription is @dataclass(eq=False) so identity-hashing makes it
             usable as a dict/set key — needed for the by-topic routing index.
           - Per-subscription max_queue override kept (snapshots are ~50 KB
             each — those subscribers will want bigger queues).
           - Closed-but-non-empty subscriptions still drain on async-for, then
             stop cleanly. Closed-and-empty raises SubscriptionClosedError.

Step 4  — Connection registry                                              [DONE]
  registry.py: tracks upstream conns, downstream consumers, ref-counted (channel, product_id) → {consumer_ids}.
  Callbacks: on_first_subscriber, on_last_unsubscriber.
  status() returns active conns, uptime, msg/sec, drop counts.
  Verify: ref-count edge-case tests pass (1→0, 0→1, churn, leaks on disconnect).
  Outcome: 32 new tests green; full suite 115/115; ruff clean.
           Decisions worth flagging:
           - Refcount keyed by full topic string (not (channel, product_id))
             so the registry stays venue-agnostic — adding binance.* later
             is just more topics, no schema change.
           - Refcount is dict[topic, set[consumer_id]] — pruned when empty so
             ``topic in _refcount`` is a true "anyone subscribed?" check.
           - Callbacks are async (Step 5 ingest will need to send WS frames).
             Fired in input order at register-time → deterministic behaviour
             for ingest implementations that batch upstream subscribe frames.
           - Callback errors propagate; we don't roll back state. The caller
             (ingest) decides whether to retry or surface the failure.
           - Direct sub.add_topic() bypasses the refcount. The registry IS the
             public API for subscription management; the bus is internal.
             Documented in the registry module docstring.
           - Consumers must go through registry; bus subs created by registry
             carry the consumer_id explicitly so logs/stats stay correlated.
           - record_message() lives on the registry rather than on the bus —
             keeps the bus a pure routing primitive; observability is hub-level
             concern. Ingest layer (Step 5) will call it on every publish.
           - msg/sec computed as messages_total / hub_uptime (a true average
             since-startup, not a sliding window). Honest and simple; can swap
             in EWMA later if dashboards demand instant rate.
           - UpstreamConnection is a small mutable dataclass owned by the
             registry; ingest will mutate state via mark_connected /
             mark_reconnecting / mark_disconnected as Step 5 lands.

Step 5  — Coinbase ingestion (real)                                         [DONE]
  ingest/coinbase.py: WS client, demand-driven sub/unsub via registry callbacks.
  Lifecycle: connect → heartbeat watchdog → exp-backoff reconnect → re-subscribe live topics.
  Normalises raw → Message envelope → publishes to bus.
  Verify: fake-WS integration test (clean connect, reconnect, stale heartbeat) + manual run vs real Coinbase.
  Outcome: 25 new tests green; full suite 140/140; ruff clean.
           Manual run vs real Coinbase: ticker / matches / level2_batch all flow
           end-to-end; 1 MB level2 snapshot parses cleanly (max_size=4 MiB fix);
           SIGINT teardown sends unsubscribe per active channel.
           Decisions worth flagging:
           - Demand model: callbacks just mutate `_desired` and signal a
             reconcile event; reconcile loop diffs `_desired` vs
             `_active_upstream` and sends sub/unsub frames grouped by channel.
             Keeps callbacks O(1) and lets the wire frames batch naturally.
           - On (re)connect we clear `_active_upstream` (server has zero subs
             after a fresh handshake) and signal reconcile → live set is
             re-subscribed in one batch per channel. Standard Coinbase recipe.
           - Two failure paths trigger reconnect: (a) WS-level errors caught in
             the run loop, (b) application-level stale feed — watchdog forces
             ws.close(code=1011) when no frame arrives within heartbeat_timeout.
             Library ping/pong only catches dead TCP; stale-but-alive needs (b).
           - Watchdog is dormant while `_desired` is empty — guards against an
             idle hub busy-looping its own connection on startup.
           - Connector is injectable (Callable[[], async-CM[WebSocketLike]]) so
             tests substitute FakeWS without monkey-patching websockets. Default
             connector is lazily-imported `websockets.connect` with max_size=4MiB.
           - Backoff is exp 1→30s, reset on successful connect. Tests pass
             tiny values (0.01s) to keep the suite under 1s.
           - Registry is attached after construction (`attach_registry`) because
             Registry takes the callbacks at __init__ → natural wiring is
             ingest → registry(callbacks=ingest.on_*) → ingest.attach_registry.
           - Control frames (subscriptions / heartbeat / error) and validation
             failures are logged + dropped — one bad frame must not kill the
             session. Receive loop is intentionally tolerant.
           - __main__ now wires bus + registry + ingest end-to-end. Operators
             can register a debug logging consumer via env var
             MDB_DEBUG_SUBSCRIBE=coinbase.ticker.BTC-USD,... for manual smoke
             tests until Step 7's WS server lands.

Step 6  — Snapshot store                                                   [DONE]
  snapshot.py: subscribes to bus internally; tracks last ticker, last trade, best bid/ask, last_update_ts per symbol.
  Verify: synthetic-stream tests pass.
  Outcome: 23 new tests green; full suite 163/163; ruff clean.
           Decisions worth flagging:
           - Snapshot store subscribes to the bus DIRECTLY, not through the
             registry. Going via the registry would create artificial demand
             and prevent ingest from ever unsubscribing upstream when real
             downstream demand drops to zero. Pinned by
             test_snapshot_store_does_not_create_registry_demand.
           - Topic tracking is dynamic (track / untrack). Wired in __main__
             by composing the registry's first/last callbacks: each transition
             fans out to BOTH ingest (drives upstream sub/unsub) and the
             snapshot store (starts/stops observing).
           - Snapshot data is RETAINED after untrack — last known state is
             still queryable while the topic is idle. last_update_at tells
             callers how stale it is. The store is a cache, not a session;
             stop()/start() preserves it too.
           - Best bid/ask sourced from Ticker (top-of-book post-trade) AND
             L2Snapshot (initial book on subscribe, sorted defensively).
             L2Update only refreshes last_update_at — full book reconstruction
             is out of scope for this hub.
           - Snapshots are keyed by product_id only. Single venue today; if a
             second venue is added that quotes the same product, the key
             should grow to (venue, product_id). Documented in the module.
           - Drain loop tolerates per-message _apply errors (logs + continues)
             — same "one bad frame must not kill the session" posture as
             ingest. Pinned by test_drain_continues_after_per_message_apply_error.
           - to_dict serialises Decimal as strings (preserves precision) and
             datetime as ISO-8601 — ready for the MCP get_snapshot tool to
             return without further conversion.
           - max_queue defaults to 4096 (vs the bus default 1000) because L2
             snapshots are ~50 KB each and arrive in bursts on (re)subscribe;
             slack avoids dropping fresh state before the drain loop catches up.

Step 7  — Downstream WebSocket server                                       [DONE]
  server/ws.py: WS server for non-MCP consumers.
  Protocol: {action: subscribe|unsubscribe, topics: [...]}.
  Each conn registered in registry, fed from bus, clean tear-down on disconnect.
  Verify: pytest test_ws_server.py passes; manual websocat shows live ticker.
  Outcome: 22 new tests green; full suite 185/185; ruff clean.
           Manual run vs real Coinbase (websocat unavailable, used a Python
           client): two concurrent clients subscribed to ticker/matches BTC-USD
           and ticker ETH-USD; live data flowed; on each client disconnect the
           hub log showed the matching ingest.unsubscribe_sent for exactly the
           topics that lost their last holder. Zero upstream subs after both
           clients gone — the no-leaks guarantee verified end-to-end.
           Decisions worth flagging:
           - Wire frames are typed by ``kind`` (welcome / ack / error / data),
             not by overloading a Coinbase-style ``type``. Keeps server-control
             messages distinct from payload-level type discriminators that
             already live inside ``data.payload.type``.
           - Each WS connection is a registry consumer, not a bus subscriber.
             Going through the registry is what makes the "disconnect →
             upstream unsubscribes if last holder" property fall out for free
             — pinned by test_disconnect_releases_topics_no_leak.
           - Topic validation happens per-topic with ``parse_topic``, and one
             bad topic in a batch returns a per-topic error frame but still
             applies the valid ones. Friendlier for clients fixing typos and
             keeps the session alive.
           - Backpressure cap (``max_dropped_messages``, default 100) is the
             "Sustained overflow → disconnect consumer with reason" hook from
             the plan. Cumulative drops > threshold → backpressure_exceeded
             error frame + close(1013). Defaults match config/default.yaml's
             bus.sustained_overflow_drops; ``None`` disables (drop-oldest only).
           - register_consumer / add_topic / remove_topic failures inside the
             recv loop are caught and surfaced as ``{action}_failed`` error
             frames with the offending topic. An ingest hiccup must not tear
             down a healthy WS session.
           - Welcome frame includes the assigned ``consumer_id`` so the client
             can correlate its own logs with hub /status output.
           - Server stop closes the listener AND every in-flight handler;
             handler ``finally`` blocks unregister consumers, which fires
             on_last_unsubscriber for any topic this client was the last
             holder of. Pinned by test_server_stop_closes_active_connections.
           - WS server starts after ingest + snapshot in __main__ — public
             surface goes up only once the inner layers are ready to accept
             demand. Stop order is the mirror: WS first, ingest, snapshot.

Step 8  — MCP server                                                       [DONE]
  server/mcp_server.py with FastMCP. Tools (LLM-first names + descriptions):
    list_topics()
    describe_topic(topic)
    get_snapshot(product_id)
    get_hub_status()
    stream_topic(topic, max_messages=100, timeout_ms=5000)   # poll-based batch
  Wired to same bus / registry / snapshot.
  Expose BOTH transports:
    `python -m market_data_broker mcp --stdio`   (default, for Claude Desktop)
    `python -m market_data_broker mcp --http`    (for Docker / remote agents)
  Verify: MCP inspector exercises every tool over stdio AND http; schemas render correctly.
  Outcome: 22 new tests green; full suite 239/239; ruff clean.
           Manual JSON-RPC framing test against stdio: initialize handshake,
           tools/list (5 tools, schemas + descriptions), tools/call list_topics
           — all returned correct structured content. HTTP transport: real
           POST against /mcp returned SSE-framed initialize reply; ingest
           connected to real Coinbase during the same boot.
           Decisions worth flagging:
           - Tool methods are also plain ``tool_*`` Python callables on the
             server class. FastMCP's @tool decorator wraps thin closures around
             them. Tests exercise the methods directly (fast, no transport)
             and verify FastMCP discovery separately via ``app.list_tools()``.
           - ``stream_topic`` registers a transient registry consumer named
             ``mcp-stream-N`` (not a bus subscription). Going through the
             registry is intentional: drives ingest sub/unsub on a 0→1 / 1→0
             transition exactly like a WS client would. The ``finally`` in
             the tool unregisters the consumer, so cancellation / exception
             paths can't leak upstream subs. Pinned by
             test_stream_topic_unregisters_on_exception.
           - Hard caps (max_messages=5000, timeout_ms=60000) clamp at
             tool-entry; we report ``truncated_*`` flags so a buggy LLM caller
             can't ask for unbounded work but still gets a response shape it
             can reason about.
           - The MCP entrypoint is a separate run mode in __main__
             (``python -m market_data_broker mcp [--stdio|--http]``). Stdio
             mode owns stdout for JSON-RPC framing — configure_logging now
             accepts a ``stream`` arg and the entrypoint passes ``sys.stderr``
             so structlog doesn't corrupt the protocol channel. ``logging.basicConfig``
             gained ``force=True`` for the same reason.
           - Hub mode and MCP mode share _build_core() (bus + ingest + snapshot
             + registry); the difference is purely which public surfaces are
             added on top. Today they run as separate processes, so a single
             box wanting both can run two compose services with different
             commands. Composing them in-process is a small extension if
             needed later — they're already isolated by transport.
           - describe_topic catalogue is inline (not file-loaded) so stdio
             deployments — where the hub is spawned by Claude Desktop and
             may not have the repo on disk — still get full schema + example
             responses.
