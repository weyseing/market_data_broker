STACK
- Python 3.11+, asyncio
- websockets, pydantic, mcp (FastMCP), structlog, pyyaml
- Tests: pytest, pytest-asyncio
- Single Docker container, in-memory state only

TOPIC NAMING
  coinbase.{channel}.{product_id}    e.g. coinbase.ticker.BTC-USD
  Channels in scope: ticker, matches, level2_batch
  Venue prefix leaves room to add binance.* later without breaking changes.

BACKPRESSURE
  Bounded asyncio.Queue per consumer (default 1000).
  Overflow → drop-oldest + increment dropped_messages counter (visible in status).
  Sustained overflow → disconnect consumer with reason.

Step 2  — Domain models & topic naming                                      [DONE]
  models.py: Pydantic Ticker, Trade, L2Update, Message envelope.
  topics.py: topic_for(), parse_topic(), validators.
  Verify: pytest tests/test_topics.py tests/test_models.py passes.
  Outcome: 54 unit tests green; lint clean (ruff E,F,I,B,UP,N).
           Decisions worth flagging:
           - Added L2Snapshot as a 4th payload type (spike showed snapshot frames
             are distinct from l2update; modelling what's on the wire beats the plan's
             3-type list).
           - Trade carries Literal["match","last_match"] — single class, both
             discriminator values, since fields are identical.
           - extra="ignore" on payload models — Coinbase adds fields without
             versioning (volume_30d precedent); tests pin the fields we depend on.
           - Exceptions named *Error to satisfy ruff N818.
           - parse_coinbase_frame() raises UnknownFrameTypeError on subscriptions /
             heartbeat / error frames; ingest layer (Step 5) must filter those before
             calling.

Step 3  — In-memory pub/sub bus                                             [DONE]
  bus.py: InMemoryBus.publish / subscribe / unsubscribe.
  Bounded per-consumer queues, drop-oldest, drop counters.
  Verify: multi-consumer fan-out + slow-consumer + dynamic sub/unsub tests pass.
  Outcome: 28 new tests green; full suite 83/83; ruff clean.
           Decisions worth flagging:
           - Subscription is the consumer handle (async-iterable, has its own
             queue, dropped_messages counter). Bus owns routing tables;
             Subscription delegates add_topic / remove_topic / close back to it.
           - publish() is async-shaped but never yields — fan-out is atomic
             w.r.t. other coroutines, so no locks. Keeps callers consistent
             (`await bus.publish(msg)`).
           - Drop-oldest implemented as get_nowait()+put_nowait() with the
             counter incremented per drop. Bus does NOT close unhealthy subs —
             that policy belongs in the registry (Step 4).
           - Subscription is @dataclass(eq=False) so identity-hashing makes it
             usable as a dict/set key — needed for the by-topic routing index.
           - Per-subscription max_queue override kept (snapshots are ~50 KB
             each — those subscribers will want bigger queues).
           - Closed-but-non-empty subscriptions still drain on async-for, then
             stop cleanly. Closed-and-empty raises SubscriptionClosedError.

Step 4  — Connection registry                                              [DONE]
  registry.py: tracks upstream conns, downstream consumers, ref-counted (channel, product_id) → {consumer_ids}.
  Callbacks: on_first_subscriber, on_last_unsubscriber.
  status() returns active conns, uptime, msg/sec, drop counts.
  Verify: ref-count edge-case tests pass (1→0, 0→1, churn, leaks on disconnect).
  Outcome: 32 new tests green; full suite 115/115; ruff clean.
           Decisions worth flagging:
           - Refcount keyed by full topic string (not (channel, product_id))
             so the registry stays venue-agnostic — adding binance.* later
             is just more topics, no schema change.
           - Refcount is dict[topic, set[consumer_id]] — pruned when empty so
             ``topic in _refcount`` is a true "anyone subscribed?" check.
           - Callbacks are async (Step 5 ingest will need to send WS frames).
             Fired in input order at register-time → deterministic behaviour
             for ingest implementations that batch upstream subscribe frames.
           - Callback errors propagate; we don't roll back state. The caller
             (ingest) decides whether to retry or surface the failure.
           - Direct sub.add_topic() bypasses the refcount. The registry IS the
             public API for subscription management; the bus is internal.
             Documented in the registry module docstring.
           - Consumers must go through registry; bus subs created by registry
             carry the consumer_id explicitly so logs/stats stay correlated.
           - record_message() lives on the registry rather than on the bus —
             keeps the bus a pure routing primitive; observability is hub-level
             concern. Ingest layer (Step 5) will call it on every publish.
           - msg/sec computed as messages_total / hub_uptime (a true average
             since-startup, not a sliding window). Honest and simple; can swap
             in EWMA later if dashboards demand instant rate.
           - UpstreamConnection is a small mutable dataclass owned by the
             registry; ingest will mutate state via mark_connected /
             mark_reconnecting / mark_disconnected as Step 5 lands.

Step 5  — Coinbase ingestion (real)                                         [DONE]
  ingest/coinbase.py: WS client, demand-driven sub/unsub via registry callbacks.
  Lifecycle: connect → heartbeat watchdog → exp-backoff reconnect → re-subscribe live topics.
  Normalises raw → Message envelope → publishes to bus.
  Verify: fake-WS integration test (clean connect, reconnect, stale heartbeat) + manual run vs real Coinbase.
  Outcome: 25 new tests green; full suite 140/140; ruff clean.
           Manual run vs real Coinbase: ticker / matches / level2_batch all flow
           end-to-end; 1 MB level2 snapshot parses cleanly (max_size=4 MiB fix);
           SIGINT teardown sends unsubscribe per active channel.
           Decisions worth flagging:
           - Demand model: callbacks just mutate `_desired` and signal a
             reconcile event; reconcile loop diffs `_desired` vs
             `_active_upstream` and sends sub/unsub frames grouped by channel.
             Keeps callbacks O(1) and lets the wire frames batch naturally.
           - On (re)connect we clear `_active_upstream` (server has zero subs
             after a fresh handshake) and signal reconcile → live set is
             re-subscribed in one batch per channel. Standard Coinbase recipe.
           - Two failure paths trigger reconnect: (a) WS-level errors caught in
             the run loop, (b) application-level stale feed — watchdog forces
             ws.close(code=1011) when no frame arrives within heartbeat_timeout.
             Library ping/pong only catches dead TCP; stale-but-alive needs (b).
           - Watchdog is dormant while `_desired` is empty — guards against an
             idle hub busy-looping its own connection on startup.
           - Connector is injectable (Callable[[], async-CM[WebSocketLike]]) so
             tests substitute FakeWS without monkey-patching websockets. Default
             connector is lazily-imported `websockets.connect` with max_size=4MiB.
           - Backoff is exp 1→30s, reset on successful connect. Tests pass
             tiny values (0.01s) to keep the suite under 1s.
           - Registry is attached after construction (`attach_registry`) because
             Registry takes the callbacks at __init__ → natural wiring is
             ingest → registry(callbacks=ingest.on_*) → ingest.attach_registry.
           - Control frames (subscriptions / heartbeat / error) and validation
             failures are logged + dropped — one bad frame must not kill the
             session. Receive loop is intentionally tolerant.
           - __main__ now wires bus + registry + ingest end-to-end. Operators
             can register a debug logging consumer via env var
             MDB_DEBUG_SUBSCRIBE=coinbase.ticker.BTC-USD,... for manual smoke
             tests until Step 7's WS server lands.
